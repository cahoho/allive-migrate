# 众生迁徙 · Allive Migration

> **一条线，一次迁徙，一个物种的未来。**
>
> 在一张持续生长的动态生态网络中，为八种真实迁徙物种规划安全路线，在与人类活动的博弈中守护生物多样性——这不是一个关于"赢"的游戏，而是一个关于"选择"的寓言。

---

## 一、作品背景

地球上每年有数以亿计的动物踏上迁徙之路：候鸟飞跃喜马拉雅，鲑鱼逆流回归产卵地，角马横渡鳄鱼密布的河流，海龟跨越大洋回到出生的海岸。然而，人类活动的扩张正在切断一条又一条生命通道——湿地消失、河流筑坝、草原围栏、海岸开发，每一处断裂都可能让一个种群走向灭绝。

**《众生迁徙》** 正是诞生于这一现实议题。作品参加腾讯云黑客松 "AI CAN DO IT" 游戏开发挑战赛，选题"小红花游戏——生物多样性与环境保护"方向，将生态学中"迁徙通道连通性"和"生物多样性韧性"的核心原理，转化为可触摸、可交互的策略游戏体验。

游戏由 **1 人 + CodeBuddy** 协作完成——从架构设计、核心逻辑、算法实现到音效音乐、概念美术，AI 全流程深度参与创作。不是用 AI 辅助写代码，而是以 AI 为创作主体，把一个想法变成一个完整可运行的游戏。

## 二、核心玩法

### 2.1 一句话概括

> 玩家在动态生成的生态地图上，通过拖拽连线为不同迁徙物种规划路线，满足各物种的栖息地需求，同时躲避人类活动威胁和自然灾害，维持生态多样性不至崩塌。

### 2.2 玩法循环

```
查看任务（物种 × 起点 → 终点 × 必经需求）
        ↓
在地图上拖拽连线，规划迁徙路线
        ↓
路线提交 → 8 层生态可行性校验
        ↓
校验通过 → 物种沿路线迁徙 → 倒计时内抵达终点 → 成功
        ↓
成功 → 揭示新节点 / 解锁新物种 / 进入下一阶段
        ↓
失败/超时 → 该物种失败计数 +1 → 累计 3 次 → 物种灭绝
        ↓
多样性 < 33.3% → 游戏结束
```

### 2.3 八大迁徙物种

游戏中没有抽象的"角色"，只有八个基于真实生态学数据设计的迁徙物种，每个物种拥有独特的栖息地需求、迁徙速度、灾害脆弱性和生态启示：

| 阶段 | 物种 | 迁徙速度 | 单次得分 | 核心需求 | 生态启示 |
|:---:|------|:---:|:---:|------|------|
| P1 | 雁鸭候鸟 | 极快 (0.55s/段) | 1.0 | 必经湿地补给站 | 湿地是候鸟长途飞行中的补给站，保护湿地能让迁徙不断线 |
| P2 | 帝王蝶 | 快 (0.85s/段) | 1.2 | 必经花蜜地 | 花蜜地像途中的能量站，连续花带越完整越容易完成迁徙 |
| P2 | 斑头雁 | 极快 (0.42s/段) | 0.9 | 必经高山通道 + 湿地补给 | 即使能飞越高山，仍依赖湿地停歇 |
| P3 | 鲑鱼 | 中等偏快 (1.25s/段) | 1.5 | 按序经过河口 → 鱼道 → 产卵地 | 河流连通是鲑鱼回家的关键 |
| P4 | 角马兽群 | 很慢 (7.20s/段) | 3.4 | 必经河流渡口 | 草场、水源和渡口共同组成迁徙廊道 |
| P5 | 美洲鳗 | 中等 (1.65s/段) | 1.7 | 必经河口（淡水→海洋） | 河流、河口和海洋是一整条生命通道 |
| P5 | 绿海龟 | 慢 (4.80s/段) | 2.7 | 礁石觅食地 → 产卵海岸 | 海岸开发会让回巢更困难 |
| P6 | 林蛙 | 极慢 (9.20s/段) | 3.8 | 必经林地缓冲带 | 小尺度林地断裂也会造成繁殖失败 |

**得分设计哲学**：越慢、越依赖连续通道的物种，生态保护价值越大，单次成功得分越高。飞行物种快速低分，陆地与两栖物种缓慢高分——这不仅是平衡设计，更是生态学原理的游戏化表达。

### 2.4 动态生态地图

地图不是静态关卡，而是一个持续生长的活的生态系统：

- **开局生成 6 个节点**，类型覆盖湿地、花蜜地、繁殖地、越冬地等，位置每局随机
- **每次成功迁徙揭示 1 个新节点**，地图最多扩展至 24 个节点
- **自动连边**：距离 ≤ 360px 的节点自动相连，每节点最多 5 条连接，保证全图单一连通分量
- **基于 mulberry32 种子随机**，每局地图完全不同，但可通过种子复现
- **所有物种共享同一张生态网络**——不存在物种专属区域，体现生态系统的互联互通

### 2.5 人类活动——现实的镜子

游戏中人类不是传统意义上的"Boss"，而是一个需要被理解、被管理的现实变量：

- **完成 3 次迁徙后激活**：先给玩家一个喘息窗口
- **单个"人类活动圈"**在地图上缓慢游走，阻挡其覆盖范围内的迁徙路线
- **鼠标可拖拽引导**人类活动圈，松手后回归屏保式自动游走
- **吞噬迁徙动物 → 圈子变大**（更难绕过）；**吞下缩圈点 → 圈子变小**（玩家可策略引导）
- **成功 12 次后进入 AI 主动追击模式**：人类活动圈智能预判动物 3.2 秒后的位置进行拦截
- **生态健康系统**：人类覆盖节点时每秒扣除 2.0 生态健康值；节点健康降至 0 则不可用，需恢复至 35 方可重新启用

### 2.6 六阶段渐进难度

| 阶段 | 触发得分 | 最大并发任务 | 任务生成间隔 | 解锁物种 |
|:---:|:---:|:---:|:---:|------|
| 1 | 0 | 1 | 12s | 雁鸭候鸟 |
| 2 | 2 | 2 | 11s | 帝王蝶、斑头雁 |
| 3 | 5 | 3 | 10s | 鲑鱼 |
| 4 | 9 | 3 | 9s | 角马兽群 |
| 5 | 13 | 4 | 8s | 美洲鳗、绿海龟 |
| 6 | 18 | 5 | 7s | 林蛙 |

从新手友好的单一任务，到硬核策略的五线并操，玩家始终处于心流区间。

### 2.7 失败条件：多样性塌缩

游戏的失败不是"死了一条命"，而是**生态系统的崩塌**：

- 同一物种失败 3 次 → 该物种**永久灭绝**，从候选池移除
- 物种多样性 = 存活物种数 / 已解锁物种数 × 100%
- **多样性 < 33.3% → 游戏结束**

这与现实生态学一致：生态系统的韧性取决于多样性，多样性的塌缩比任何一次单独失败都更致命。

## 三、CodeBuddy 创作历程

### 3.1 一个人 + CodeBuddy = 一个完整的游戏

《众生迁徙》的开发是一场人机协作的实验：**1 个人，1 个 AI 搭档，约 2 周开发周期，产出 50+ 源文件、5000+ 行 TypeScript 代码、20+ AI 生成音效/BGM、6 张 AI 概念美术图。**

CodeBuddy 不是单纯的代码补全工具，而是贯穿全流程的创作伙伴：

```
想法萌芽 → 架构设计 → 类型定义 → 系统实现 → 算法开发 → Bug 修复 → 构建部署
    ↓          ↓          ↓          ↓          ↓          ↓          ↓
 人类描述    CodeBuddy    CodeBuddy    CodeBuddy    CodeBuddy    CodeBuddy    CodeBuddy
 需求       设计架构     定义类型     实现逻辑     编写算法     定位修复     部署验证
```

### 3.2 CodeBuddy 驱动的代码开发

**技术栈**：Vue 3 + Composition API + Vite 5 + TypeScript（严格模式）+ SVG + CSS

**架构设计**：CodeBuddy 协助完成了清晰的分层架构：

| 层级 | 模块 | 说明 |
|------|------|------|
| 表现层 | 15 个 Vue 组件 | TopBar、GameMap、SpeciesPanel、EventOverlay、GameOverModal 等 |
| 状态层 | gameStore.ts (~1900 行) | 基于 Vue reactive 的自研状态管理，全局状态 + 所有操作入口 |
| 系统层 | 6 个独立模块 | 种子随机、地图生成、图生成、任务生成、路线求解、世界事件 |
| 组合层 | 3 个 Composables | 游戏主循环 (RAF)、拖线交互、路线验证 |
| 数据层 | 配置 + 模板 | 全局常量、节点模板、8 种物种模板、事件定义 |

**核心算法（均由 CodeBuddy 实现或辅助实现）**：

- **种子随机系统**：mulberry32 算法，保证同一种子可复现同一局地图
- **动态地图生成**：节点坐标生成 + 距离约束 + 连通性保证
- **图生成与连边**：动态连边 + 度数限制 + 单一连通分量验证
- **任务生成**：起点/终点/需求抽取 + 多解保证（BFS/DFS 预检至少 1 条合法路径）
- **路线求解**：BFS 最短路 + DFS 多解 + 需求校验 + 资源成本计算
- **8 层路线验证**：起终点校验 → 节点可用性 → 边连通性 → 无重复 → 生态需求 → 灾害影响 → 资源预算
- **世界事件系统**：5 种灾害的三阶段生命周期（预警 → 激活 → 恢复）+ 公平性预检（模拟事件生效后确保所有任务仍有合法路径）
- **人类活动 AI**：屏保式游走 + 鼠标拖拽引导 + 预测追击（3.2 秒前瞻 + 采样步长 0.25s）
- **生态健康系统**：人类扣血 + 迁徙承载压力 + 延迟恢复 + 状态分级（healthy → pressured → damaged → degraded）

### 3.3 AI 全能力参与

除代码开发外，AI 还参与了游戏的音频与视觉创作：

**自适应音乐系统（9 层）**：
- 主旋律层（L0）：马林巴 + 钢琴 + 竖琴 + 弦乐，始终播放，象征生态链本身
- 8 个物种轨：每个物种解锁时对应乐器淡入，灭绝时淡出
- 物种 × 乐器映射：长笛=候鸟、钟琴=蝴蝶、短笛=斑头雁、大提琴=鲑鱼、中提琴=角马、低音单簧管=美洲鳗、钢片琴=海龟、卡林巴琴=林蛙
- 6 种互动机制：物种解锁叠加、灾害不协和音、阶段调性上移、灭绝静默、终局淡出、暂停衰减
- 所有音轨由 WorkBuddy 的 game-audio-generator skill 调用 AI 模型生成

**概念美术**：
- 游戏主视觉图 100% AI 生成（Text-to-Image）
- 8 个物种 SVG 图标先由 AI 生成概念稿，再转化为矢量代码
- UI 色彩方案和布局草案由 AI 辅助产出

### 3.4 后端与反作弊

游戏使用 Supabase（PostgreSQL + Edge Functions）提供在线排行榜功能，设计了 **6 层反作弊验证**：

1. **HMAC-SHA256 令牌签名**：会话令牌由服务端密钥签名，客户端无法伪造
2. **令牌过期验证**：令牌有时效，超时自动失效
3. **令牌去重（防重放）**：同一令牌只能提交一次，数据库标记 used 状态
4. **得分边界验证**：根据物种得分表 + 生存时间计算理论最大值，超出即拒绝
5. **生存时间合理性**：非零得分要求至少 5 秒生存时间
6. **得分/时间组合验证**：交叉校验得分与时间的合理性

**离线模式**：未配置 Supabase 时游戏完全可用，仅排行榜功能不可用——零配置门槛，开箱即玩。

## 四、作品亮点

### 亮点一：生态叙事 × 策略玩法的深度融合

不是在策略游戏上"贴一层环保皮"，而是把生态学原理写成游戏机制：

- **迁徙通道连通性** → 地图节点与连边的动态生成
- **栖息地特异性** → 每个物种独特的 tag 需求（湿地/花蜜/河口/鱼道/渡口/林地）
- **生物多样性韧性** → 多样性 < 33.3% 即游戏结束
- **人类活动的生态压力** → 人类活动圈扣减节点生态健康
- **灾害的系统性影响** → 暴风/干旱季节性切换，影响不同物种的可通行区域

玩家在游戏中做出的每一个决策——选哪条路线、优先保护哪个物种、是否引导人类活动圈远离关键节点——都是真实生态保护中会面临的取舍。

### 亮点二：纯 Web 原生，零依赖运行

- **Vue 3 + SVG + CSS**：无 Canvas、无 Phaser、无 Unity、无 PNG/JPG/外部字体
- **浏览器原生运行**：一键部署至 CloudStudio，任意设备打开即玩
- **严格 TypeScript**：类型安全，工程级代码质量
- **响应式交互**：Pointer Events 同时支持鼠标和触摸，桌面/移动端无缝适配

### 亮点三：AI 全流程驱动的创作范式

| 创作环节 | 传统方式 | 本作方式 |
|----------|----------|----------|
| 代码架构 | 人工设计 | CodeBuddy 协作设计 + 实现 |
| 核心算法 | 人工编写 | CodeBuddy 实现 BFS/DFS/种子随机等 |
| 背景音乐 | 委托作曲 / 素材库 | AI 模型生成 9 层自适应音乐 |
| 音效设计 | 素材库采购 | AI 生成 20+ 物种/交互/灾害音效 |
| 概念美术 | 委托画师 | AI Text-to-Image 生成主视觉 |
| 视觉图标 | 设计师制作 | AI 概念稿 → 矢量代码转化 |

**从"写代码"到"写 Prompt"——AI 不只是工具，是创作的主体。**

### 亮点四：严谨的工程实现

- **8 层路线验证**：每一次路线提交都经过"生态可行性审查"
- **公平性预检**：世界事件激活前模拟所有任务的可解性，确保不会产生死局
- **6 层反作弊**：HMAC 签名 + 令牌去重 + 得分边界 + 时间合理性
- **种子可复现**：同一种子生成同一局地图，便于调试和竞技
- **渐进式难度曲线**：并发任务数、生成间隔、时间倍率三个维度平滑递进

### 亮点五：自适应音乐——可听化的生态系统

音乐不是背景噪音，而是游戏状态的可听化表达：

- 物种解锁 → 对应乐器淡入，玩家"听到"生态变丰富
- 物种灭绝 → 乐器淡出 + 短暂静默，玩家"听到"失去了什么
- 灾害激活 → 不协和音程 + 低频轰鸣，玩家"听到"危险来临
- 阶段跃迁 → 调性上移半音 + BPM 微升，玩家"听到"压力增大
- 终局将至 → 所有装饰轨淡出只剩主旋律，玩家"听到"生态系统在崩塌

## 六、总结

《众生迁徙》试图回答一个问题：**当 AI 可以从零构建一个完整游戏时，人类创作者的角色是什么？**

答案是：**人类负责"为什么"——为什么候鸟需要湿地，为什么多样性重要，为什么人类不是反派而是现实。AI 负责"怎么做"——怎么生成动态地图，怎么验证路线合法性，怎么让音乐跟随游戏状态变化。**

每一条成功绘制的路线，都是对这个星球的一次温柔守护。而这个游戏本身，也是一次关于人与 AI 如何协作守护创造力的实验。
